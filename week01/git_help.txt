💡 Git 快捷操作小抄（零基础版）
1️⃣ 保存文件

在 VS Code 里修改文件后 Ctrl + S 保存到本地硬盘

这是最基础的一步，Git 只管文件内容，不管你在编辑器里怎么改

2️⃣ 查看修改状态
git status


查看哪些文件被修改或新增

红色 → 修改未暂存

绿色 → 已暂存准备提交

3️⃣ 暂存修改（add）
git add .


. 表示把所有修改都暂存

也可以只暂存部分文件：

git add 文件名


作用：告诉 Git “我想把这些修改保存到下一次提交里”

4️⃣ 提交修改（commit）
git commit -m "写一句说明，比如：更新学习笔记"


作用：生成快照，把暂存区的修改归档到 Git 历史

建议 commit 信息简短、清楚说明修改内容

示例：

git commit -m "修正RK3588学习笔记错误"

5️⃣ 上传到 GitHub（push）
git push origin main


将本地 commit 上传到远程仓库

第一次 push 会要求输入 GitHub 用户名 + Token

成功后刷新 GitHub 网页就能看到更新

6️⃣ 小技巧

每次修改前拉取最新远程更改（避免冲突）：

git pull origin main


只想上传部分文件：

git add 文件名
git commit -m "说明"
git push origin main


查看提交历史：

git log


查看修改内容差异：

git diff

🔹 简单记忆顺序
修改文件 → git add → git commit → git push

git add .	“保存当前全部游戏进度”	将工作区中所有已修改/新增的文件，一次性全部放入暂存区。
git add 文件名	“只保存某个支线任务的进度”	只将指定的一个或多个文件放入暂存区，其他改动保留在工作区。


例子：
根据你之前“事件驱动”的思路，我为你总结一个傻瓜式操作口诀：

“一查，二选，三存档。”

一查 (git status)：
动手前先看“清单”。终端会告诉你哪些文件被改了（红色），哪些已经在“准备区”（绿色）。这是你做出选择的基础。

二选 (选择 add 的策略)：

如果红字文件全是关于同一件事的 -> 用 git add .（全部打包）。

如果红字文件混杂了不同的事（例如既有代码a.cpp，又有文档README.md） -> 用 git add a.cpp 和 git add README.md 分开添加。

三存档 (git commit -m “...”)：
为你刚刚选中的、已进入“准备区”的改动，创建一个正式的存档点。

# 1. 先存档事件A：修复
git add hello.cpp
git commit -m “fix: 修复了hello.cpp中的空指针错误”

# 2. 再存档事件B：文档
git add README.md
git commit -m “docs: 更新学习日记，记录修复bug的过程”

# 3. 最后一起推送到云端
git push
